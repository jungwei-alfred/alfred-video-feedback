規格說明

1常駐回饋列（在原「回報」位置）

把「問句 + 👍 / 👎」放在播放器底部（原回報入口處），無論播放 / 暫停 / 播完都常駐顯示。

  • 跟播放控制列綁在一起：控制列 3 秒後自動隱藏時一起消失；用戶點影片叫回控制列時，這排也一起回來。
  • A 版（原始）對應位置放的是「回報」按鈕；B 版放 👍 / 👎。

2問句與 👍 / 👎

量「這次偵測判斷對不對」（準確度，不是價值感）。

  • 問句：固定為「這次有抓對嗎？」（等同「Did we get it right?」），不依 event type 變化。按鈕標示「正確 / 不正確」。
  • UI：單色 outline（描邊）拇指圖示，去掉黃色 emoji 的娛樂感。
  • 對比：控制列 / 回饋列下方加由下往上的漸層遮罩＋白字陰影——直式影片（相機直立、畫面填滿）或亮背景時也清楚，不會被影片內容吃掉。
  • 送出：點 👍/👎 開對應 sheet，關閉後才送出；一支影片一次、送出後整排回饋列消失（避免看得到卻不能改的挫敗感）。

3送出後整列消失

給完回饋即視為確實送出、一支影片一次定論。送出後整排回饋列直接消失，不留下「看得到卻不能改」的鎖定狀態，避免挫敗感。

  • 事件在關閉 bottom sheet 後才送出（見「資料寫入與埋點」）；以 toast 確認已送出。

4點 👎 → 自動開回報 bottom sheet

倒讚為負向訊號；同時自動帶出回報表單，讓想說明原因的人無縫接續。不填也沒關係，👎 已記錄。事件於關閉表單後送出。

5授權鎖（回報表單）

詳細回報仍須勾選授權才能送出；未勾選時「送出」維持 disabled。讚 / 倒讚不受此限。「其他」可開放式填答（須勾選「其他」展開輸入框，內容也計入可送出條件）。

「其他」字數上限 100 字，對齊 Firebase Analytics 事件參數值上限（100 半形字元）；接近上限（≥80）才顯示計數器，滿 100 轉紅。

6👍 後跳「同意分享」dialog

點 👍 即選取正向訊號（送出後整列消失），隨即彈出影片分享詢問「願意分享這支影片，協助我們優化偵測品質嗎？」，用滿意當下的時機爭取分享。

  • 「同意分享 / 下次再說」雙鍵，明確主動同意（不預設）。
  • 左上 X 或點背景遮罩皆可關閉，👍 已記錄、不受分享與否影響；無「選填」字樣。
  • 僅保留一行範圍說明（僅這支、僅工程團隊）。

授權 vs 互動對照

互動出現時機需影片授權送出方式
👍 讚常駐（底部，隨控制列隱藏）免授權選取 → 開分享 dialog → 關閉時送出
👎 倒讚常駐（底部，隨控制列隱藏）免授權選取 → 開回報 → 關閉時送出
💬 回報A 版入口（隨控制列隱藏）需授權選項/其他 + 勾選授權

待討論決策點

事件發送時機（已更新）

本版採：無論 👍 或 👎，點擊只負責「選取 + 開啟 bottom sheet」，事件一律在使用者關閉 sheet 後才發送（送出、同意分享、下次再說、X、點遮罩都算關閉）。
好處：一次寫入完整事件（含分享與否 / 回報內容），避免「先送一筆、後補一筆」的重複或拆裂；不填也仍會送出 emoji 訊號。

常駐回饋列的干擾

問句 + 👍/👎 常駐(跟控制列一起 3 秒自動隱藏)。每支都顯示是否造成疲勞 / 誤觸，待觀察；可評估抽樣或只前幾支顯示。

送出後整列消失

一支影片一次定論；送出後整排回饋列消失（不留鎖定狀態）。是否需要可改 / 收回再議。

👍 後跳授權 dialog 的摩擦取捨

權衡：👍 本是最低門檻的正向動作，剛給好評就被要影片授權，可能有「給讚反被索取」的觀感、降低 👍 完成率。務必維持「明顯可關閉、👍 已先記錄」。
替代：只對部分流量 / 前幾次 👍 才跳；或不每支都問。值得 A/B 看授權同意率 vs 👍 完成率。

👍/👎 語意維度

本版定為：偵測是否正確，問句固定「這次有抓對嗎？」（放棄 event type 差異）。量的是準確度而非價值感——「有沒有幫助」無法對應到判斷對錯。

埋點（第一天就上）

事件類型、相機型號、偵測類型、App 版本、時間 + 是否經強調觸發 + 變體（A / B）。事件於關閉 sheet 時寫入，附帶：分享與否、回報選項、其他文字（≤100 字）。

A/B 測試規格

假設

在現況（「回報」按鈕）之外，於播放器底部常駐 👍 / 👎 偵測正確性詢問，能用更低門檻、更顯眼的入口收到更多回饋，且不犧牲回報品質。上方工具列可切 A / B 實際比較。

變體

A 對照組：「回報」按鈕（現況）。
B 實驗組：播放器底部常駐「這次有抓對嗎？＋ 👍 / 👎」（👎 帶出回報、👍 帶出分享）。兩者都跟控制列一起常駐 / 自動隱藏。

主要指標

  • 入口點擊率（CTR）：點擊回饋入口的觀看數 ÷ 回放觀看總數。A = 點「回報」；B = 點 👍 / 👎。
  • 回報成功率：完成送出回報 ÷ 開啟回報表單。

次要指標

每次觀看的回饋率、👍:👎 比、影片分享同意率、「其他」填寫率、誤觸 / 取消率。

分流與期間

使用者層級隨機分派，同一使用者固定看同一變體；跑到統計顯著或至少 1–2 週（涵蓋週末）。

注意

B 多了純 👍/👎 這個 A 沒有的入口，總回饋量本就會偏多——比較時要把「👍/👎」與「回報」分開定義事件，並看回報品質是否維持。

1. 背景

F-Squad 六月目標之一：「雙平台 75% 以上的設備使用 YOLOv26n，剩餘低階裝置則使用小 model 來運作 Person Detection」。完成後的下一步，是要決定 AI model on SW Camera 之後要怎麼走——因此需要更多來自用戶的 input，方便後續決策。

2. 問題

主要問題

  • 我們現在無法得知 Premium 用戶對目前 AI model 滿不滿意，只有相對間接的指標可參考（PD 開啟、功能留存率，或 PD 用戶的訂閱率、訂閱留存率等）。
  • 用戶回饋的入口藏得太深：目前兩個入口都在 Event Player 內——(1) 右上 ⋯ More 裡的「提供影片反饋」（藏得較深）；(2) 影片暫停時右下角的「回報」（需符合特定條件才出現）。
  • 現在的回報內容沒有上報到 Firebase、只有打 API，所以想前後比較「回報」的點擊率也做不到——因為沒有點擊數，只有實際回報的數量。

次要問題

  • 用戶必須同意影片授權才能送出回饋，可能減少影片回收的數量。nice-to-have

3. 解決方案方向

  • 在 Event Player 上提供讚 / 倒讚，並提供一個開放式 feedback 管道。
  • 新埋點都放到 Firebase（含讚、倒讚、feedback 內容），方便後續數據分析；同時保留 API 寫入。

4. 目標與非目標

目標

  • 提供更直覺、低門檻的回饋入口，提高回饋率，補足「Premium 用戶對 AI model 滿意度」的直接訊號。
  • 用 👍 / 👎 取得可看大盤趨勢的滿意度訊號（免影片授權）。
  • 負向情境順勢收集可診斷的細節（回報表單 + 開放填答）。
  • 新事件上 Firebase，補足「回報入口點擊率」這類目前缺的數據。
  • 以 A/B 驗證新入口優於現況，再決定全量。

非目標

  • 不在本期重構 More / 既有深層回報入口（保留為備援）。
  • 不為了提高影片授權率而使用預設打勾等 dark pattern。

5. 成功指標

  • 主要：入口點擊率（CTR）、回報成功率（完成送出 ÷ 開啟表單）。
  • 次要：每次觀看回饋率、👍:👎 比、影片分享同意率、「其他」填寫率、誤觸/取消率。

6. 使用者故事

  • 身為使用者，看完一段回放後，我想用一鍵 👍 / 👎 快速表達好不好，不必填表單。
  • 身為使用者，當影片有問題時，我想順手選問題類型、必要時補一句說明。
  • 身為 PM，我想看大盤滿意度趨勢，並能切維度找出哪類事件 / 機型滿意度偏低。

8. 資料寫入與埋點

  • 雙寫：讚 / 倒讚與 bottom sheet（回報 / 分享）都同時打 API 與 Firebase 事件（讚、倒讚、feedback 內容都要進 Firebase）。
  • 發送時機：Firebase 事件於 bottom sheet 消失後發送，一次寫入完整事件。
