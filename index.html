規格說明

1位置：疊在播放器右下角（原「回報」入口處）

👍 / 👎 放在播放畫面右下角，即原本「回報」按鈕出現的位置，沿用既有入口的直覺。

  • 跟播放控制列一起 auto-hide：播放時隱藏、暫停 / 結束時顯示 → 按「暫停/看完」觀察。
  • 權衡：貼近原入口、較直覺，但會稍微遮到畫面，且播放時看不到（與控制列連動）。

2暫停 / 結束時強調

在播放暫停或結束那一刻，對 👍 / 👎 做一次微動畫（nudge），精準捕捉「看完」的時間點。

用「暫停/看完」按鈕觀察按鈕 pulse 動畫。

2讚 / 倒讚：按過即鎖定，slim bar 消失

給完回饋即視為確實送出，不可切換或反悔——已選 highlight 後整條 slim bar 收合消失。

  • 目的：避免用戶切來切去，難以判斷真實評價；一支影片一次定論。
  • 消失前以 toast 確認（「感謝你的回饋」），讓用戶知道已送出。
  • 單純「回報」（未按 emoji）則須等表單實際送出後 bar 才消失；中途關閉表單，bar 保留。

4點 👎 → 送出 + 自動開回報 bottom sheet

倒讚本身即視為送出（大盤負向訊號），slim bar 隨即收合；同時自動帶出回報表單，讓想說明原因的人無縫接續。不填也沒關係，👎 訊號已記錄。

已移除 slim bar 上的獨立「回報」按鈕——回報表單改由 👎 觸發（或右上 ⋯ 的既有深層入口），回饋列只留 👍 / 👎 更精簡。

5授權鎖（回報表單）

詳細回報仍須勾選授權才能送出；未勾選時「送出」維持 disabled。讚 / 倒讚不受此限。「其他」可開放式填答（須勾選「其他」展開輸入框，內容也計入可送出條件）。

「其他」字數上限 100 字，對齊 Firebase Analytics 事件參數值上限（100 半形字元）；接近上限（≥80）才顯示計數器，滿 100 轉紅。

6👍 後跳「同意分享」dialog

點 👍 即送出正向訊號（slim bar 收合），隨即彈出影片分享詢問「願意分享這支影片，協助我們優化偵測品質嗎？」，用滿意當下的時機爭取分享。

  • 「同意分享 / 下次再說」雙鍵，明確主動同意（不預設）。
  • 左上 X 或點背景遮罩皆可關閉，👍 已記錄、不受分享與否影響；無「選填」字樣。
  • 僅保留一行範圍說明（僅這支、僅工程團隊）。

授權 vs 互動對照

互動需影片授權送出方式送出後
👍 讚免授權選取 → 開分享 dialog → 關閉時才送出鎖定，bar 消失
👎 倒讚免授權選取 → 開回報 → 關閉時才送出鎖定，bar 消失
💬 回報需授權選項/其他 + 勾選授權送出後 bar 消失

待討論決策點

事件發送時機（已更新）

本版採：無論 👍 或 👎，點擊只負責「選取 + 開啟 bottom sheet」，事件一律在使用者關閉 sheet 後才發送（送出、同意分享、下次再說、X、點遮罩都算關閉）。
好處：一次寫入完整事件（含分享與否 / 回報內容），避免「先送一筆、後補一筆」的重複或拆裂；不填也仍會送出 emoji 訊號。

強調時機

建議：暫停 + 播放結束都觸發一次強調；避免重複強調造成干擾（同一支只 nudge 一次）。

誤觸無法挽回

鎖死 + bar 消失代表誤觸即定論。可評估誤觸率後，再決定是否要加回「收回」緩衝或其他補救機制。

👍 後跳授權 dialog 的摩擦取捨

權衡：👍 本是最低門檻的正向動作，剛給好評就被要影片授權，可能有「給讚反被索取」的觀感、降低 👍 完成率。務必維持「明顯可關閉、👍 已先記錄」。
替代：只對部分流量 / 前幾次 👍 才跳；或不每支都問。值得 A/B 看授權同意率 vs 👍 完成率。

👍/👎 語意維度

建議：收斂為單一維度（影片是否有幫助），引導語明確，否則大盤難解讀。

埋點（第一天就上）

事件類型、相機型號、偵測類型、App 版本、時間 + 是否經強調觸發 + 變體（A / B）。事件於關閉 sheet 時寫入，附帶：分享與否、回報選項、其他文字（≤100 字）。

A/B 測試規格

假設

把入口從單一「回報」按鈕（A）換成 👍 / 👎（B），能降低回饋門檻、收到更多回饋，且不犧牲回報品質。上方工具列可切 A / B 實際比較。

變體

A 對照組：播放器右下角僅一顆「回報」按鈕（現況入口）。
B 實驗組：同位置改為 👍 / 👎（👎 帶出回報、👍 帶出分享）。

主要指標

  • 入口點擊率（CTR）：點擊回饋入口的觀看數 ÷ 回放觀看總數。A = 點「回報」；B = 點 👍 或 👎。
  • 回報成功率：完成送出回報 ÷ 開啟回報表單。A 由「回報」進入；B 由 👎 進入。

次要指標

每次觀看的回饋率、👍:👎 比、影片分享同意率、「其他」填寫率、誤觸 / 取消率。

分流與期間

使用者層級隨機分派，同一使用者固定看同一變體；跑到統計顯著或至少 1–2 週（涵蓋週末）。

注意

兩版「入口」的語意不同（A 只有回報；B 多了純 emoji），CTR 不能直接劃等號——需各自定義入口事件，並把「總回饋量」與「回報量」分開看。

1. 背景與問題

目前影片回饋的入口都在 Event Player 內，且都偏深：(1) 右上 ⋯ More 裡的「提供影片反饋」（藏得深）；(2) 影片暫停時右下角的「回報」（需符合特定條件才出現）。點進去後只有可複選的問題選項、沒有開放式填答，且都必須勾選影片授權才能送出。

結果是回饋門檻高、樣本少，也缺少一個能看「整體滿意度趨勢」的輕量訊號。

2. 目標與非目標

目標

  • 提供更直覺、低門檻的回饋入口，提高回饋率。
  • 用 👍 / 👎 取得可看大盤趨勢的滿意度訊號（免影片授權）。
  • 負向情境順勢收集可診斷的細節（回報表單 + 開放填答）。
  • 以 A/B 驗證新入口優於現況，再決定全量。

非目標

  • 不在本期重構 More / 既有深層回報入口（保留為備援）。
  • 不為了提高影片授權率而使用預設打勾等 dark pattern。

3. 成功指標

  • 主要：入口點擊率（CTR）、回報成功率（完成送出 ÷ 開啟表單）。
  • 次要：每次觀看回饋率、👍:👎 比、影片分享同意率、「其他」填寫率、誤觸/取消率。

4. 使用者故事

  • 身為使用者，看完一段回放後，我想用一鍵 👍 / 👎 快速表達好不好，不必填表單。
  • 身為使用者，當影片有問題時，我想順手選問題類型、必要時補一句說明。
  • 身為 PM，我想看大盤滿意度趨勢，並能切維度找出哪類事件 / 機型滿意度偏低。

5. 功能需求

5.1 入口位置與顯示

👍 / 👎 疊在播放畫面右下角（即原「回報」按鈕位置），跟播放控制列一起 auto-hide：播放時隱藏、暫停 / 結束時顯示。暫停或結束那一刻對按鈕做一次 nudge 微動畫，捕捉「看完」時機。

5.2 讚 / 倒讚行為

  • 點擊只負責「選取（highlight）+ 開啟對應 bottom sheet」；單選互斥。
  • 事件一律在使用者關閉 sheet 後才發送（送出、同意分享、下次再說、X、點遮罩皆算關閉），一次寫入完整事件。
  • 送出後鎖定、按鈕淡出消失（一次定論，不可反悔）。
  • 👍 / 👎 免影片授權。

5.3 👎 → 回報表單

👎 帶出回報 bottom sheet：可複選問題類型；「其他」展開開放式填答，上限 100 字（對齊 Firebase Analytics 事件參數值上限），接近上限（≥80）才顯示計數器、滿 100 轉紅。詳細回報須勾選影片授權才能送出。

5.4 👍 → 同意分享 dialog

👍 帶出影片「同意分享」dialog：「願意分享這支影片，協助我們優化偵測品質嗎？」配「同意分享 / 下次再說」雙鍵（不預設），含一行範圍說明（僅這支、僅工程團隊），左上 X 與點遮罩皆可關閉。分享與否不影響 👍 已被記錄。

5.5 備援入口

右上 ⋯ More 的「提供影片反饋」保留為既有深層入口。

6. 埋點規格

事件於關閉 sheet 時寫入，至少包含：事件類型、相機型號、偵測類型、App 版本、時間、是否經強調觸發、變體（A / B），並依情境附帶：分享與否、回報選項、其他文字（≤100 字）。

7. A/B 測試計畫

假設：把入口從單一「回報」按鈕（A）換成 👍 / 👎（B），能降低回饋門檻、收到更多回饋，且不犧牲回報品質。

主要指標：入口 CTR、回報成功率。分流：使用者層級隨機，同一使用者固定看同一變體。期間：跑到統計顯著或至少 1–2 週（涵蓋週末）。

注意：兩版入口語意不同（A 只有回報、B 多了純 emoji），CTR 不能直接劃等號——需各自定義入口事件，並把「總回饋量」與「回報量」分開看。

8. 待討論決策點

  • 事件發送時機：本版採「關閉 sheet 後發送」。
  • 誤觸無法挽回：鎖死 + 消失，是否視誤觸率加回「收回」緩衝。
  • 👍 後跳分享 dialog 的摩擦：是否只對部分流量 / 前幾次顯示。
  • 👍 / 👎 語意維度：建議收斂為「影片是否有幫助」。
  • 影片授權：是否值得另做提高同意率的優化（拆開、contextual、文案）。
